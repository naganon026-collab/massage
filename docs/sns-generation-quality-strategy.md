# 高品質SNS生成を実現するシステム・プロンプト戦略

> 美容院SNSマーケティング戦略ガイドを参照し、凡庸なAI出力から脱却するための設計・実装指針

---

## 1. 現状の課題と「凡庸さ」の根本原因

### 1-1. プロンプト設計上のボトルネック

| 原因 | 説明 | 結果 |
|------|------|------|
| **単一ステップ生成** | 役割・コンテキスト・制約・出力を1つの長いシステムプロンプトに詰め込み、モデルに「一発で完結文」を書かせている | モデルが「無難な平均」に収束し、具体性・意外性が消える |
| **抽象指示の過多** | 「自然に織り込む」「丁寧に」「親しみやすく」など、評価基準が曖昧な表現が多い | 解釈の幅が広く、教科書的な「です・ます調の説明文」になりやすい |
| **汎用フォールバックの強さ** | 「髪質改善」「トレンドカラー」「丁寧なカウンセリング」を強みとして毎回固定で指示している | 店舗ごとの差が消え、どの店でも同じ言い回しになる |
| **具体例の不足** | ガイドにある「NGヘアケア3選」「施術データ（所要時間・仕上げ）」のような**型**をプロンプト内で明示していない | 出力が「髪でお悩みの方へ…」レベルの一般論に留まる |
| **コンテンツ柱との紐づけ不足** | ビフォーアフター / 教育 / スタッフ / 裏側の「4つの柱」や、投稿タイプ別のテンプレートをプロンプトが参照していない | 目的（保存率狙い vs 新規認知）に応じた構成・トーンが出ない |

### 1-2. システムフロー上のボトルネック

- **入力の粗さ**: ユーザーが選ぶ「パターン」と3問の回答だけでは、ガイドで求められる「施術名・所要時間・価格目安」「問題→原因→解決策の3ステップ」「保存・フォロー・予約の3つCTA」を満たす**具体情報**が不足しがち。
- **コンテキストの単一化**: 店舗の「その他特記事項」「スクレイピング内容」があっても、長大なシステムプロンプトの一部として埋め込まれるだけなので、**どの段落で何を優先すべきか**がモデルに伝わりにくい。
- **検証ルールの不在**: 生成後に「フックが1〜2行か」「CTAに予約導線が含まれるか」「ハッシュタグが3層構造か」などをチェックするステップがなく、品質がぶれやすい。

これらが重なり、「なんとなく正しいが、誰が書いても似たような文章」＝凡庸な出力になっている。

## 2. 添付ファイルから読み解く「売れるコンテンツ」の要件

### 2-1. 構造的価値（ガイドの骨格）

- **4つの柱の明示**: ①ビフォーアフター ②プロの知識・教育 ③スタッフの属人性 ④サロンの裏側・日常。投稿ごとに「どの柱か」が決まっており、目的・推奨頻度・具体例がセットになっている。
- **3ステップの厳守**: フック（1〜2行で止める）→ ボディ（リスト＋理由＋解決策）→ CTA（保存・フォロー・予約の明示）。ガイドはこれを「黄金テンプレート」として全投稿共通にしている。
- **プラットフォーム別の役割分担**: フィード＝信頼・ポートフォリオ、リール＝新規・バイラル、ストーリーズ＝リピート・関係強化。同一本文でも、媒体ごとに「何を最優先するか」が定義されている。

### 2-2. 具体性の源泉

- **数字・固有名詞**: 「NGヘアケア3選」「180度以上」「8割乾かしてから就寝」「約3時間」「グラデーションカラー」など、再現可能なレベルまで落ちている。
- **施術データの明記**: テンプレートBでは「施術・所要時間・仕上げ」「お悩み」「担当からのコメント」がブロック分けされており、**何を書くべきか**が型として固定されている。
- **保存率・指標の言及**: 「保存数÷リーチ数＞3%」「プロフィール遷移率＞5%」など、運用目標が書かれており、コンテンツの「狙い」が明確。

### 2-3. トーン＆マナー

- 専門用語は「理由＋解決策」とセットで使う（例：硫酸系成分を避けたアミノ酸系シャンプー＋なぜか）。
- 「保存して見返してください」「プロフィールのリンクから○○をご予約ください」など、**次のアクションが一文で分かる**。
- 絵文字・ハッシュタグは「親しみやすさ」と「検索・保存」の両立用にルール化されている。

### 2-4. 実務直結のアクションプラン

- 週次コンテンツカレンダー（曜日×柱）、PDCAの確認指標、継続投稿の3大ルール（撮影習慣化・制作タイム確保・ネタ帳）など、**明日から使える手順**が書かれている。

これらを「プロンプトやデータとしてシステムに落とし込むべき要件」として整理すると、次の戦略に繋がる。

## 3. 高品質な出力を実現するためのシステム＆プロンプト戦略

### 3-1. 単一プロンプト依存からの脱却

- **タスクの細分化**: 「1回の生成で全媒体・全文を書く」のではなく、  
  - ステップ1: 今回の投稿の「柱」と「目的（保存率 vs 新規認知）」を決定  
  - ステップ2: フック案を3案生成し1つ選択（またはユーザー選択）  
  - ステップ3: ボディを「問題→原因→解決策」のブロックで生成  
  - ステップ4: CTA・ハッシュタグを媒体別に付与  
  のように分けると、各ステップで「型」を強制しやすい。
- **マルチステップ生成の実装**: バックエンドで順次APIを叩き、前ステップの出力を次ステップのコンテキストに渡す。ユーザーには「一括生成」のように見せつつ、内部では2〜4回の短いプロンプトに分割する。

### 3-2. ガイドの「型」をRAG・静的コンテキストで注入

- **RAGの対象**: ガイド本文をチャンク分割し、「4つの柱」「テンプレートA/B」「プラットフォーム別指針」「週次カレンダー」などを検索可能にする。生成時に「今回のパターン＝教育系」なら「柱②＋テンプレートA」のチャンクを優先してコンテキストに含める。
- **静的プロンプトへの組み込み**: ガイドの「投稿テンプレートA/B」の見出しとブロック構成（【フック】【ボディ】【CTA】と、ボディ内の「❶ 〜 → 理由 → 解決策」）を、そのままシステムプロンプトの「出力フォーマット」に書き、**思考プロセスでまずこの型を埋めさせる**。

### 3-3. Few-Shotの戦略的配置

- **柱・パターン別の1本ずつ**: 教育系・ビフォーアフター・スタッフ紹介・サロン裏側のそれぞれについて、ガイドに近い「完成投稿1本」をサンプルとして保持し、システムプロンプトまたはユーザープロンプトに「以下の例の構成・トーン・具体性のレベルに合わせて書くこと」と含める。
- **店舗別サンプル**: ユーザーが「過去の投稿サンプル」を登録している場合は、それを強み・文体のFew-Shotとして優先し、汎用の「髪質改善・トレンドカラー」より前に配置する。

### 3-4. ペルソナ・目的の明示

- ガイドの「対象：地域密着型サロン（スタッフ3〜5名）」「ターゲット：20〜30代女性」を、システムプロンプトの冒頭で毎回固定する。
- 加えて「今回の投稿の目的」を1行で渡す（例：「保存率を狙う教育系」「新規認知を狙うビフォーアフター」）。これで、同じパターンでも「保存推奨」と「予約誘導」でトーン・CTAを切り替えられる。

### 3-5. 出力検証（ポストプロセス）の導入

- 生成JSONに対し、簡易ルールで検証する層を設ける。  
  - フックが2行以内か  
  - CTAに「予約」「プロフィール」「リンク」のいずれかが含まれるか  
  - ハッシュタグ数が指定範囲か  
- 不満足なら「CTAを追加して再生成」など、短いプロンプトで補正する。ユーザー体験としては「生成→表示」のままでよく、内部だけ2段階にできる。

## 4. SaaSバックエンド用：最適化されたシステムプロンプト実装例

以下は、**教育系（柱②）・保存率狙い**の1投稿を想定した、そのまま組み込んでテストできるシステムプロンプトのひな形である。役割・コンテキスト・制約・出力型・思考プロセスを分離している。

```markdown
# 役割
あなたは美容院SNSマーケティング戦略ガイドに準拠した「教育系コンテンツ」のライターです。20〜30代女性が「保存したくなる」投稿を、ガイドの「テンプレートA」の型で作成します。

# 今回の対象店舗（必ずこの情報のみ使用すること）
- 店舗名：{{shopName}}
- 住所：{{shopAddress}}
- 業種：{{shopIndustry}}
- 強み・独自情報：{{shopFeatures}}
- WEB参考情報：{{shopScrapedContent}}

# 投稿の目的（この1本の狙い）
- 保存数増加・専門家ブランディング。リーチ拡大。
- 「保存して見返してください」のCTAを必ず含む。

# 必須の構成（ガイド 4-2 テンプレートAに厳守）
1. 【フック】最初の1〜2行。以下のいずれかの型で書く。
   - 「○選」型：例「現役美容師が絶対やらないNGヘアケア3選」
   - 「疑問」型：例「なぜあなたの髪はすぐパサつくのか」
   - 「数字・保存」型：例「保存必須！髪質改善の新常識」
2. 【ボディ】リスト形式。各項目を「❶ 問題 → 理由（なぜダメか）→ 解決策」の3点セットで書く。2〜3行以内で簡潔に。
3. 【CTA】次の2つを両方含める。
   - 予約誘導：「プロフィールのリンクから○○をご予約ください」と具体的に。
   - 保存促し：「この投稿、後で見返すために『保存』もお忘れなく」など。

# 制約
- 店舗名「{{shopName}}」を本文中に1回以上自然に含める。
- 上記「強み・独自情報」「WEB参考情報」がある場合は、その表現を優先して使い、汎用の「髪質改善・トレンドカラー」は補助に留める。
- 専門用語を使う場合は「理由＋解決策」をセットで書く。抽象的な「〜です」「〜できます」のみで終えない。

# 思考プロセス（内部で必ず実行し、出力には含めない）
1. 今回のテーマに合う「フックの型」（○選 / 疑問 / 保存必須）を1つ選ぶ。
2. ボディのリスト項目を3つ決め、各項目の「問題・理由・解決策」を1文ずつメモする。
3. そのメモに沿って、テンプレートAの見出し【フック】【ボディ】【CTA】の順で本文を書く。
4. 最後にハッシュタグを、ビッグ3・ミドル7・スモール5の3層で15個付ける。

# 出力形式
JSONのみ。装飾なし。
{"instagram": "【フック】\n...\n【ボディ】\n...\n【CTA】\n...\n\n#タグ ...", "gbp": "...", "portalTitle": "...", "portal": "...", "line": "..."}
```

- `{{shopName}}` 等はリクエスト時の変数で置換する。
- パターンが「ビフォーアフター」の場合は、上記の「必須の構成」をガイドのテンプレートB（施術データ・お悩み・担当コメント・CTA）に差し替えた別システムプロンプトを用意する。

### 4-2. マルチステップ用：フック生成専用プロンプト（ステップ1）

```markdown
# 役割
教育系投稿の「フック」（最初の1〜2行）のみを3案作成する。

# 条件
- テーマ：{{userTheme}}（ユーザー入力またはパターンから決定）
- 型：「○選」「疑問」「保存必須」のいずれかを使う。
- 20〜30代女性がスクロールを止める、具体的で意外性のある一文にすること。

# 出力形式
JSONのみ。
{"options": ["案1の全文", "案2の全文", "案3の全文"]}
```

このあと、選ばれた1案をユーザープロンプトに含めて「ボディ＋CTA」を生成する2ステップ目を叩く設計にできる。

## 5. 開発に向けたネクストアクション

| 優先度 | アクション | 担当・目安 |
|--------|------------|------------|
| 1 | **ガイドの構造をコードに落とす** | 4つの柱・テンプレートA/Bのブロック名と項目を定数（またはDB）で定義し、`patternId` に応じて「使用するテンプレートID」を切り替える。既存の `generate/route.ts` の `patternId` 分岐に「テンプレートA/B」のフォーマット文字列を追加する。 |
| 2 | **システムプロンプトの先頭を「今回の対象店舗」で固定** | すでに「【重要】今回の対象店舗」を追加済みなら維持。未実施なら、店舗名・住所・強みを最初のブロックに置き、その直後に「上記以外を使ってはいけない」を1行入れる。 |
| 3 | **教育系・B/A用のFew-Shotを1本ずつ用意** | ガイドのテンプレートA/Bの完成例をそのまま1投稿分コピーし、`LP_DEMOS` や別JSONファイルで保持。生成時に `patternId` が B なら教育系サンプル、A ならB/Aサンプルをシステムプロンプトに追加する。 |
| 4 | **「投稿の目的」をリクエストに含める** | 既存の `patternTitle` / `patternId` に加え、`purpose: "save" | "new_customer" | "repeat"` のような1フィールドを渡し、プロンプト内の「今回の狙い」で参照する。UIではパターン選択と連動して自動設定してもよい。 |
| 5 | **ボディを2ステップ生成にする（任意）** | まず「フック3案」を生成→1つ選ぶ（または先頭採用）→そのフックをコンテキストに含めて「ボディ＋CTA」を生成、という2回呼び出しを試す。レイテンシとコストが許す範囲で導入。 |
| 6 | **出力検証レイヤーの追加** | 生成JSONの `instagram` 文字列に対し、「プロフィール」「予約」「リンク」のいずれかを含むか、正規表現でチェック。含まれなければ `CTAを追加して30字以内で補足し、同じJSONキーで返す` という短いプロンプトを1回だけ叩く。 |
| 7 | **RAGの検討（中長期）** | ガイドPDFや社内ナレッジをチャンク化し、`patternId` や `purpose` に応じて検索→上位3チャンクをコンテキストに付与。まずは静的プロンプトの拡張で対応し、必要になった段階でRAGを導入する。 |

## 中長期：ガイド本文のRAG検討

以下の設計で、必要になった段階でRAGを導入する。

### 対象ドキュメント
- 美容院SNSマーケティング戦略ガイド（PDF/テキスト）
- 4つの柱・テンプレートA/B・プラットフォーム別指針・週次カレンダー・PDCA指標など

### チャンク設計
- 見出し単位で分割（例：「柱① ビフォーアフター」「テンプレートA：教育系」「3-1. Instagram」）
- 1チャンク 200〜400 トークン程度。メタデータに `pillar` / `templateId` / `platform` を付与

### 検索トリガー
- リクエストの `patternId` と `purpose` からクエリを生成  
  例：patternId=B, purpose=save → 「教育系 保存率 リスト フック CTA」
- ベクトル検索（またはキーワード）で上位 3〜5 チャンクを取得

### コンテキスト注入
- 取得チャンクをシステムプロンプトの「【参照ガイド抜粋】」として挿入
- 静的プロンプト（CONTENT_TEMPLATES / FEW_SHOT）を優先し、RAGは補足に留める

### 技術候補
- 埋め込み: Vertex AI Text Embedding または OpenAI Embeddings
- ストア: Supabase pgvector または メモリ/ファイルキャッシュ（小規模なら）
- チャンク化: 自前スクリプト or LangChain / LlamaIndex のテキストスプリッター

### 運用
- ガイド改訂時にチャンクの再生成・再投入が必要
- まずは静的定数＋Few-Shotで品質を固め、「型は揃ったがバリエーションが足りない」段階でRAGを検討する

## まとめ

- **凡庸さの原因**は、単一プロンプト・抽象指示・汎用フォールバック・具体例不足・ガイドの型の未反映にある。
- **ガイドの価値**は、4つの柱・3ステップ構成・テンプレートA/B・プラットフォーム別役割・数字と固有名詞の具体性にある。
- **戦略**は、タスク細分化・マルチステップ、ガイドの型のRAG/静的注入、Few-Shot、目的・ペルソナの明示、出力検証の組み合わせで実現する。
- **実装**は、上記のシステムプロンプトひな形を `generate/route.ts` のパターン別分岐に組み込み、続けてFew-Shotと目的フィールドを追加するところから着手できる。

このドキュメントをベースに、まずは「教育系」と「ビフォーアフター」の2パターンでプロンプトとFew-Shotを強化し、出力を比較検証することを推奨する。
